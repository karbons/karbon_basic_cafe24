# 3. 다국어 URL 정책 (v2.0 동일)

모든 서비스 공통:

```
/{appType}/{lang}/{...}
```

예)

```
/main/ko/auth/login
/app/en/dashboard
/admin/ko/building/list
```

# 4. 언어 기본값 처리 정책 (업데이트)

언어는 다음 우선순위로 적용된다:

### ➤ 1순위: URL prefix (`/ko` → ko)

### ➤ 2순위: 쿠키 (`lang=ko`)

### ➤ 3순위: 브라우저 언어

### ➤ 4순위: 기본값 `ko`

# 5. 언어 쿠키 저장 기능

### 5.1 쿠키 스펙

| 키      | 값            | 설명        |
| ------ | ------------ | --------- |
| `lang` | `ko`, `en` 등 | 선택된 언어 저장 |

### 5.2 쿠키 설정 정책

* 경로: 도메인 전체(`/`)
* 만료: 1년
* Secure = true, SameSite=Lax
* HttpOnly = false (JS에서 사용해야 하므로)

### 5.3 SvelteKit 구현

```ts
// src/hooks.server.ts
import { redirect } from '@sveltejs/kit';

export function handle({ event, resolve }) {
  const { cookies, url } = event;

  if (!url.pathname.startsWith('/main') &&
      !url.pathname.startsWith('/app') &&
      !url.pathname.startsWith('/admin')) {
    return resolve(event);
  }

  const segments = url.pathname.split('/');
  const appType = segments[1];
  const lang = segments[2];

  if (!['ko', 'en'].includes(lang)) {
    const saved = cookies.get('lang') ?? 'ko';
    throw redirect(302, `/${appType}/${saved}`);
  }

  cookies.set('lang', lang, {
    path: '/',
    maxAge: 60 * 60 * 24 * 365,
    sameSite: 'lax',
    secure: true
  });

  return resolve(event);
}
```

# 6. 언어 선택 Dropdown 기능

### 6.1 기본 원칙

언어 변경 시:

* 쿠키 변경
* URL도 변경
* 페이지 전체 reload 필요 (i18n lazy load 때문)

### 6.2 Svelte 컴포넌트 예시

```svelte
<script>
  import { locale } from 'svelte-i18n';
  import { base } from '$app/paths';
  export let currentLang;

  const changeLang = (code) => {
    locale.set(code);
    const newPath = window.location.pathname.replace(`/${currentLang}/`, `/${code}/`);
    window.location.href = newPath;
  };
</script>

<select bind:value={currentLang} on:change={() => changeLang(currentLang)}>
  <option value="ko">한국어</option>
  <option value="en">English</option>
</select>
```

# 7. 백엔드 API 언어 처리 (확장됨)

## 7.1 요청 규칙

프론트 → API

```http
Accept-Language: ko
```

## 7.2 백엔드 공통 추출 로직

언어 우선순위:

1. API 헤더 Accept-Language
2. 쿠키 lang
3. 기본값 ko

### Rust (Axum)

```rs
let lang = headers
    .get("Accept-Language")
    .and_then(|v| v.to_str().ok())
    .unwrap_or("ko");
```

# 8. API i18n 메시지 매핑 테이블 (신규)

## 8.1 목적

* 에러 메시지/성공 메시지/시스템 메시지를 프론트와 백엔드가 동일하게 공유
* 백엔드는 key만 응답하고, 프론트에서 번역 처리 가능
* 또는 백엔드에서 바로 메시지를 번역해서 전달 가능 (선택식)

---

## 8.2 메시지 키 규칙

```
{도메인}.{기능}.{메시지명}
```

예)

```
auth.login.invalid
auth.login.success
building.create.success
building.delete.fail_used
errors.permission_denied
```

---

## 8.3 표준 매핑 테이블

| Key                        | ko                    | en                             |
| -------------------------- | --------------------- | ------------------------------ |
| `auth.login.invalid`       | 아이디 또는 비밀번호가 잘못되었습니다. | Invalid username or password.  |
| `auth.login.success`       | 로그인 성공                | Login successful               |
| `errors.permission_denied` | 권한이 없습니다.             | Permission denied.             |
| `building.create.success`  | 건물이 생성되었습니다.          | Building created successfully. |

> 이 테이블은 `/locales/*.json`과 동일한 구조로 관리됨

---

## 8.4 백엔드 응답 전략

### 전략 A — 백엔드에서 번역된 문자열을 응답 (권장)

응답 예시:

```json
{
  "success": false,
  "message": "아이디 또는 비밀번호가 잘못되었습니다.",
  "code": "auth.login.invalid"
}
```

프론트: 그대로 출력

---

### 전략 B — 백엔드에서 code만 주고 프론트에서 번역하는 방식

응답:

```json
{
  "success": false,
  "code": "auth.login.invalid"
}
```

프론트:

```svelte
{$t(response.code)}
```

**장점:**

* 번역 관리가 프론트에 집중됨
* 언어 전환 시 API 재응답 필요 없음

| -------- | ------------------------- |
